# F1TENTH Debugging & Troubleshooting

이 문서는 F1TENTH ROS2 자율주행 프로젝트를 구현하면서 발생한 문제와 해결 과정을 정리한 문서입니다.

---

## 1. 자동 긴급 정지 알고리즘 오작동

### 문제 상황

LiDAR 기반 자동 긴급 정지 기능을 구현한 뒤, 실제 충돌 위험이 크지 않은 상황에서도 차량이 갑자기 정지하는 문제가 발생했습니다.

특히 차량이 벽 옆을 지나가거나 코너를 통과할 때, 전방 장애물이 아닌 측면 벽을 충돌 위험으로 판단하여 불필요하게 제동하는 경우가 있었습니다.

### 원인

초기 구현에서는 LiDAR 전체 beam 중 가장 가까운 장애물 하나만 기준으로 TTC(Time-To-Collision)를 계산했습니다.

하지만 F1TENTH 차량 기준으로 전방, 좌측, 우측의 위험도는 서로 다릅니다.

- 전방 장애물: 실제 충돌 가능성이 높음
- 측면 벽: 차량이 평행하게 지나가면 충돌 가능성이 낮음
- 후방 또는 비진행 방향: 긴급 정지 판단에서 제외 가능

또한 LiDAR 데이터에 포함될 수 있는 `inf`, `nan`, 0에 가까운 비정상 값이 TTC 계산에 영향을 줄 수 있었습니다.

### 해결

LiDAR beam을 차량 기준 구역으로 나누어 처리했습니다.

```text
전방 영역: 충돌 위험에 민감하게 반응
좌측/우측 영역: 완화된 기준 적용
불필요 영역: 판단 대상에서 제외
```

추가로 LiDAR 값 전처리를 적용했습니다.

- `nan` 값 제거
- `inf` 값 최대 거리로 대체
- 너무 작은 거리값 필터링
- 차량 속도가 0에 가까울 때 TTC 계산 제한

### 결과

불필요한 급정지가 줄었고, 실제 전방 충돌 위험이 큰 상황에서만 안정적으로 정지할 수 있었습니다.

## 2. LiDAR beam 각도 처리 오류

### 문제 상황

장애물 방향을 기준으로 전방, 좌측, 우측을 구분했지만, 실제 주행에서는 차량이 예상과 다른 방향으로 회피하는 문제가 발생했습니다.

예를 들어 오른쪽 장애물을 왼쪽 장애물처럼 판단하거나, 전방 장애물의 각도가 어긋나 조향 방향이 반대로 계산되는 경우가 있었습니다.

### 원인

LaserScan 메시지는 단순 거리 배열이 아니라, `angle_min`, `angle_increment`, `ranges`를 함께 이용해야 실제 각도를 계산할 수 있습니다.

초기에는 ranges 배열 index만 기준으로 구역을 나눴기 때문에 실제 차량 기준 각도와 코드상 index가 정확히 일치하지 않았습니다.

### 해결

각 beam의 실제 각도를 다음 방식으로 계산했습니다.

```text
angle = angle_min + index * angle_increment
```

이후 계산된 angle을 기준으로 전방, 좌측, 우측 영역을 다시 정의했습니다.

RViz에서 `/scan` 데이터를 확인하면서 코드상 구역과 실제 장애물 위치가 일치하는지도 검증했습니다.

### 결과

장애물 방향 판단이 안정화되었고, 회피 조향 방향이 실제 환경과 일치하게 되었습니다.

## 3. Follow-the-Gap에서 좁은 통로를 주행 가능 영역으로 판단

### 문제 상황

장애물 회피 알고리즘을 구현한 뒤, 차량이 지나가기 어려운 좁은 통로를 빈 공간으로 판단하는 문제가 발생했습니다.

LiDAR 거리값만 보면 양쪽 벽 사이에 빈 공간이 존재하지만, 실제 차량의 폭을 고려하면 충돌 가능성이 있는 구간이었습니다.

### 원인

초기 Follow-the-Gap 알고리즘은 단순히 임계값보다 큰 연속 거리 구간을 gap으로 판단했습니다.

이 방식은 다음 요소를 고려하지 못했습니다.

- 차량의 실제 폭
- 회전 시 필요한 여유 공간
- 장애물 주변 안전 반경
- LiDAR 중심점과 차량 body 크기의 차이

### 해결

가장 가까운 장애물 주변에 Safety Bubble을 적용했습니다.

```text
가장 가까운 장애물 탐색
→ 해당 index 주변 일정 범위 제거
→ 남은 free space에서 가장 긴 gap 탐색
→ gap 중앙 방향으로 조향
```

또한 너무 좁은 gap은 차량이 통과할 수 없는 영역으로 판단하여 후보에서 제외했습니다.

### 결과

차량 중심점 기준이 아니라 차량 body와 안전 반경을 고려한 회피가 가능해졌습니다. 좁은 통로로 무리하게 진입하는 문제가 줄었습니다.

## 4. 장애물 회피 중 조향값이 흔들리는 문제

### 문제 상황

Follow-the-Gap 주행 중 차량이 좌우로 흔들리는 문제가 발생했습니다.

LiDAR 값이 프레임마다 조금씩 바뀌면서 선택되는 best point가 계속 달라졌고, 그 결과 조향각도 빠르게 변했습니다.

### 원인

매 프레임 가장 넓은 gap을 즉시 선택했기 때문에 센서 노이즈가 그대로 조향 명령에 반영되었습니다.

또한 조향각 변화량에 제한이 없어 이전 조향각과 현재 조향각의 차이가 커질 수 있었습니다.

### 해결

LiDAR ranges에 smoothing filter를 적용했습니다.

```text
kernel_size = 5
moving average filter 적용
```

또한 조향각 크기에 따라 속도를 다르게 설정했습니다.

```text
조향각 작음 → 직선 구간 → 속도 증가
조향각 큼 → 회전 구간 → 속도 감소
```

### 결과

차량의 지그재그 움직임이 줄었고, 장애물 회피 중 안정성이 개선되었습니다.

---

## 5. Wall Following에서 벽과의 거리 유지가 불안정한 문제

### 문제 상황

PID 기반 Wall Following을 구현했지만, 벽과의 거리가 일정하게 유지되지 않고 차량이 벽에 가까워졌다 멀어졌다를 반복하는 문제가 있었습니다.

### 원인

PID gain이 주행 환경과 차량 속도에 맞지 않았습니다.

- `kp`가 너무 크면 조향이 과하게 반응
- `kd`가 너무 작으면 오차 변화에 둔감
- `ki`가 너무 크면 누적 오차로 인해 overshoot 발생

또한 고정 속도로 주행하면 코너 구간에서 벽과의 오차가 급격히 커졌습니다.

### 해결

PID gain을 조정하고, 조향각 크기에 따라 속도를 조절했습니다.

```text
조향각 0~10도: 고속
조향각 10~20도: 중간 속도
조향각 20도 이상: 저속
```

### 결과

벽과의 거리 오차가 줄었고, 코너 구간에서도 차량이 더 안정적으로 벽을 따라갈 수 있었습니다.

## 6. Pure Pursuit에서 waypoint를 놓치는 문제

### 문제 상황

Pure Pursuit 경로 추종 중 차량이 waypoint를 정확히 따라가지 못하고 코너에서 경로 밖으로 벗어나는 문제가 발생했습니다.

특히 속도가 빠르거나 lookahead distance가 적절하지 않을 때 문제가 두드러졌습니다.

### 원인

Pure Pursuit는 lookahead distance 설정에 따라 주행 특성이 크게 달라집니다.

- lookahead가 너무 짧음: 조향이 민감해져 흔들림 발생
- lookahead가 너무 김: 코너에서 충분히 꺾지 못함

또한 차량 뒤쪽에 있는 waypoint를 선택하면 이미 지나간 지점을 다시 따라가려는 문제가 발생할 수 있습니다.

### 해결

차량 좌표계 기준으로 waypoint가 전방에 있는지 검사했습니다.

```text
x_vehicle_frame > 0 인 waypoint만 후보로 사용
```

그리고 조향각을 차량의 최대 조향각 범위로 제한했습니다.

```text
max_steering = 0.4189 rad
```

### 결과

차량이 뒤쪽 waypoint를 목표로 삼는 문제가 줄었고, 코너 구간에서 경로 추종 안정성이 개선되었습니다.

## 7. Occupancy Grid 좌표 변환 오류

### 문제 상황

RRT* 경로 계획 중 노드가 주행 가능 영역에 생성되어야 하는데, RViz에서 보면 벽 안쪽이나 장애물 영역에 생성되는 문제가 있었습니다.

반대로 실제로는 주행 가능한 공간인데 충돌 영역으로 판단되는 경우도 있었습니다.

### 원인

map 좌표계와 Occupancy Grid index 좌표계 변환이 정확하지 않았습니다.

Occupancy Grid는 다음 정보를 함께 고려해야 합니다.

- map origin
- resolution
- width / height
- image 좌표계와 ROS map 좌표계 방향 차이

초기에는 world coordinate와 grid index 변환 과정에서 origin과 resolution 처리가 정확하지 않아 좌표가 어긋났습니다.

### 해결

world 좌표를 grid index로 변환할 때 origin과 resolution을 명확히 반영했습니다.

```text
grid_x = int((world_x - origin_x) / resolution)
grid_y = int((world_y - origin_y) / resolution)
```

또한 map image를 읽을 때 좌표계 방향 차이를 보정했습니다.

### 결과

RRT* node와 edge가 실제 주행 가능 영역 기준으로 생성되었고, RViz 시각화 결과와 충돌 검사 결과가 일치하게 되었습니다.

## 8. 장애물 근처에 경로가 생성되는 문제

### 문제 상황

RRT*가 생성한 경로가 장애물과 너무 가까워 실제 차량 크기를 고려하면 충돌할 가능성이 있었습니다.

지도상으로는 경로가 장애물 cell과 겹치지 않았지만, 차량 body 기준으로는 벽에 닿을 수 있는 경로였습니다.

### 원인

초기 충돌 검사는 차량 중심점 기준으로만 수행되었습니다.

즉, 경로상의 점이 장애물 cell과 겹치지 않으면 안전하다고 판단했지만, 실제 차량은 폭과 길이가 있기 때문에 중심점만으로는 충분하지 않았습니다.

### 해결

Occupancy Grid에서 장애물 영역을 확장했습니다.

```text
원본 obstacle cell
→ 차량 크기만큼 inflation
→ inflated obstacle 영역도 충돌 영역으로 판단
```

또한 edge collision check에서 시작점과 끝점만 검사하지 않고, 두 노드 사이의 중간 지점도 일정 간격으로 검사했습니다.

### 결과

장애물에 너무 가까운 경로가 제거되었고, 차량 크기를 고려한 안전한 경로가 생성되었습니다.

---

## 9. RRT* 경로 생성 시간이 오래 걸리는 문제

### 문제 상황

RRT*는 랜덤 샘플링 기반이기 때문에 복잡한 구간에서 목표 지점까지 경로를 찾는 데 시간이 오래 걸렸습니다.

특히 좁은 통로나 코너가 많은 구간에서는 불필요한 node가 많이 생성되어 실시간 주행 적용이 어려웠습니다.

### 원인

초기에는 전체 map 영역에서 무작위로 sample을 생성했습니다.

이 방식은 탐색 범위가 넓고, goal 방향으로 tree가 확장된다는 보장이 없어 목표 도달까지 시간이 오래 걸립니다.

### 해결

다음 방법으로 탐색 효율을 개선했습니다.

- sampling 영역 제한
- goal bias 적용
- 유효하지 않은 node 즉시 제거
- Occupancy Grid 기반 빠른 collision check
- goal과 가까운 방향으로 tree 확장 유도

### 결과

불필요한 node 생성이 줄었고, 목표 방향으로 tree가 확장되는 비율이 증가했습니다.

---

## 10. RRT* 경로가 차량이 따라가기 어려운 방향으로 생성되는 문제

### 문제 상황

RRT*가 생성한 path가 좌표상으로는 연결되지만, 실제 Ackermann 차량이 따라가기 어려운 급격한 방향 전환을 포함하는 경우가 있었습니다.

### 원인

RRT*는 기본적으로 점과 점을 직선으로 연결하는 방식입니다.

하지만 F1TENTH 차량은 differential drive 로봇처럼 제자리 회전이 불가능하고, 조향각 제한을 가진 Ackermann 구조입니다.

따라서 heading 변화가 너무 큰 경로는 실제 차량이 추종하기 어렵습니다.

### 해결

parent node와 child node 사이의 heading 차이를 계산하고, 일정 각도 이상 차이가 나는 후보는 제외했습니다.

또한 node 간 거리와 조향 제한을 함께 튜닝하여 Pure Pursuit가 추종 가능한 경로가 생성되도록 조정했습니다.

### 결과

급격한 방향 전환이 줄었고, 경로 추종 중 차량이 path를 이탈하는 문제가 감소했습니다.

---

## 11. 동적 장애물 대응 속도 문제

### 문제 상황

RRT*로 생성된 경로를 따라가는 중 갑자기 장애물이 나타나면, 경로를 새로 생성하는 데 시간이 걸려 즉각적인 대응이 어려웠습니다.

### 원인

RRT*는 cost 계산과 rewiring을 포함하기 때문에 단순 회피 알고리즘보다 계산량이 많습니다.

동적 장애물이 갑자기 등장했을 때 RRT*만으로 즉시 대응하기에는 한계가 있었습니다.

### 해결

경로 계획과 긴급 안전 제어를 분리했습니다.

```text
일반 주행: RRT* path + Pure Pursuit
위험 상황: Safety Bubble + Emergency Stop 우선 처리
```

즉, 기본 주행은 RRT* 경로를 따르되 가까운 장애물이 감지되면 LiDAR 기반 안전 로직이 우선 작동하도록 구성했습니다.

### 결과

전체 경로 계획의 안정성과 실시간 장애물 대응을 분리할 수 있었습니다.

---

## 12. RViz 시각화와 실제 경로가 어긋나는 문제

### 문제 상황

RViz에서 보이는 RRT tree, waypoint, 차량 위치가 실제 주행 위치와 어긋나 보이는 문제가 있었습니다.

### 원인

Marker 메시지의 `frame_id`와 경로 생성에 사용한 좌표계가 일치하지 않았습니다.

map, odom, base_link 좌표계가 혼재되면 RViz에서 시각화가 실제 위치와 다르게 표시됩니다.

### 해결

RRT tree, path, waypoint marker의 `frame_id`를 경로 생성 기준 좌표계와 일치시켰습니다.

또한 Odometry에서 얻은 차량 pose와 Occupancy Grid 기준 좌표가 동일한 좌표계에서 해석되는지 확인했습니다.

### 결과

RViz에서 보이는 경로와 실제 차량 위치가 일치하게 되었고, 디버깅 과정에서 경로 생성 결과를 더 정확히 확인할 수 있었습니다.

## 13. 절대경로로 인한 실행 환경 의존 문제

### 문제 상황

다른 PC에서 실행할 경우 waypoint 파일과 map 파일을 찾지 못하는 문제가 발생할 수 있었습니다.

코드에 다음과 같은 절대경로가 포함되어 있었기 때문입니다.

```text
/home/yoon/sim_ws/log/waypoint.csv
/home/yoon/sim_ws/log/wp-2025-05-14-09-31-38.csv
~/sim_ws/src/f1tenth_gym_ros/maps/levine.yaml
```

### 원인

개발 중 빠른 테스트를 위해 사용자 PC 기준 경로를 직접 코드에 작성했습니다.

하지만 GitHub에 공개하거나 다른 환경에서 실행하려면 경로를 일반화해야 합니다.

### 해결 방향

다음과 같은 방식으로 개선할 수 있습니다.

- ROS2 parameter로 waypoint_path, map_path 입력
- `config/*.yaml`로 경로 관리
- `get_package_share_directory()`로 패키지 내부 파일 참조
- README에 실행 전 경로 수정 안내 추가

### 결과

현재 README에는 실행 전 경로 수정이 필요하다는 내용을 명시했습니다. 향후 개선 시 parameter 기반 구조로 바꾸면 재사용성이 높아집니다.

## 14. GitHub 업로드 시 중첩 Git 저장소 문제

### 문제 상황

`f1tenth_gym_ros` 폴더 내부에 `.git` 폴더가 포함되어 있으면 상위 프로젝트에 GitHub 업로드 시 해당 폴더가 submodule처럼 처리되거나 비어 보일 수 있습니다.

### 원인

`f1tenth_gym_ros`가 외부 저장소에서 clone된 패키지이기 때문입니다.

### 해결

GitHub 업로드 전 내부 `.git` 폴더를 제거합니다.

```bash
rm -rf src/f1tenth_gym_ros/.git
```

### 결과

상위 프로젝트의 일반 폴더로 정상 업로드할 수 있습니다.
